# Database Reliability & Persistence Protocols

Velarix is designed for reasoning integrity in regulated environments. Data loss is prevented through a multi-layered durability strategy.

## 🛡️ Core Durability Guarantees
- **Sync-on-Write**: Every fact assertion or state change is synchronized to the physical disk (`WithSyncWrites(true)`) before the API returns a success code. This protects against system crashes and power failures.
- **Journaled Persistence**: All operations are recorded in a high-performance value log (LSM tree). This provides an immutable audit trail of every state transition.
- **Atomic Operations**: State updates are performed within BadgerDB transactions, ensuring no partial or corrupted states are ever committed.

## 🔄 Recovery & Startup
- **Hybrid Boot Strategy**: 
    1. The engine first loads the most recent binary **Snapshot**.
    2. It then replays only the trailing **Journal Entries** from the disk that occurred after the snapshot.
- **Impact**: This ensures near-instant startup even with millions of historical facts, without sacrificing consistency.

## 💾 Backup Mechanisms

### 1. Automated Daily Backups
The server runs an internal ticker that triggers a full database backup every **24 hours**.
- **Location**: Backups are saved as `velarix_backup_{timestamp}.bak` in the working directory.
- **Safety**: These backups are point-in-time consistent even while the server is under load.

### 2. Manual Admin Backups
Administrators can trigger or fetch a backup at any time via the API:
- **Endpoint**: `GET /v1/org/backup`
- **Integrity**: Each backup is hashed using **SHA-256**. The hash is recorded in the Admin Audit Trail for compliance verification.

### 3. Disaster Recovery (Restore)
A backup can be restored to a fresh or existing instance:
- **Endpoint**: `POST /v1/org/restore` (Accepts the `.bak` binary stream).
- **Process**: This replaces the current engine state with the state contained in the backup.

## 🗄️ Storage Best Practices for Production
- **Encryption**: Always set `VELARIX_ENCRYPTION_KEY` to enable AES-256 encryption-at-rest.
- **External Volume**: Map the `velarix.data` directory to a persistent cloud volume (EBS, Azure Managed Disk, etc.) to survive container restarts.
- **Off-site Backups**: Periodically move the `.bak` files generated by the automated ticker to a separate storage bucket (S3, GCS).
